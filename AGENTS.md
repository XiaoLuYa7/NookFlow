# Agent.md

## 项目说明

本项目是一个 macOS 灵动岛应用，界面和交互应符合 macOS 设计习惯，整体效果需要自然、流畅、简洁，并尽量贴近 Apple 原生应用体验。

## 开发原则

### 1. 性能优先

开发过程中必须重点关注性能，避免不必要的资源消耗。

- 严格控制内存占用。
- 严格控制 CPU 占用。
- 禁止无意义的高频轮询。
- 禁止创建无法正确释放的定时器、监听器、任务或对象。
- 页面隐藏、窗口关闭或功能停用后，必须停止对应任务并释放资源。
- 动画应保持流畅，但不能通过持续高刷新率或高频计算换取视觉效果。
- 非必要情况下不要持续刷新整个界面。
- 优先采用事件驱动、按需更新和增量更新。
- 避免重复读取文件、重复请求数据和重复计算。
- 大量数据处理不得阻塞主线程。
- 后台任务完成后，涉及 UI 的更新必须切回主线程。
- 使用缓存前必须评估缓存大小、生命周期和清理策略。

### 2. 代码简洁

代码应清晰、直接、容易维护。

- 优先使用简单可靠的实现，不进行无意义的过度封装。
- 方法、类型和变量命名必须准确表达用途。
- 单个方法只处理一个明确职责。
- 避免过深的嵌套和过长的方法。
- 避免重复代码，但不要为了消除少量重复而设计复杂抽象。
- 不引入没有实际必要的第三方依赖。
- 修改功能时尽量控制影响范围。
- 不随意重构与当前任务无关的代码。
- 删除废弃代码、无效注释和调试输出。
- 注释应说明设计原因和特殊约束，不要重复描述代码本身。

### 3. 需求不明确时先询问

遇到以下情况时，不允许直接猜测实现：

- 需求存在多种合理理解。
- 交互方式没有明确说明。
- 修改可能影响现有功能。
- 涉及文件移动、删除、覆盖或权限操作。
- 涉及订阅、付费、用户数据或隐私。
- 无法确定目标 macOS 版本。
- 无法确定应该使用 SwiftUI、AppKit 或混合实现。
- 无法确认视觉效果、动画时长或交互优先级。
- 无法确认修改范围。

必须先向用户提出简洁、具体的问题，确认后再继续实现。

不要一次提出大量问题，只询问会直接影响实现结果的关键点。

## 技术要求

### 1. macOS 开发

- 优先使用 Apple 官方框架。
- 优先遵循 macOS Human Interface Guidelines。
- SwiftUI 能稳定实现的功能优先使用 SwiftUI。
- 涉及精细窗口控制、拖拽、全局事件、复杂滚动或高性能列表时，可以使用 AppKit。
- SwiftUI 无法可靠实现时，不要强行使用复杂绕过方案，应考虑使用 AppKit 封装。
- 注意多屏幕、刘海屏、不同缩放比例和不同分辨率。
- 注意窗口层级、焦点、鼠标事件、键盘事件和辅助功能权限。
- 不得依赖固定屏幕坐标完成核心逻辑。

### 2. 灵动岛窗口

- 灵动岛窗口应保持轻量。
- 窗口不应持续触发无效布局。
- 收起状态下应停止不需要的动画和数据刷新。
- 不可见内容不得持续更新。
- 动画开始前取消冲突动画。
- 避免频繁创建和销毁窗口，优先复用窗口实例。
- 窗口尺寸变化应平滑，避免抖动、跳帧和布局闪烁。
- 弹出、展开、收起和切换状态时，应明确管理状态，避免多个状态同时生效。
- 非必要情况下不要抢占用户当前应用的输入焦点。

### 3. 并发与任务管理

- 不在主线程执行耗时操作。
- 所有异步任务都应有明确的生命周期。
- 可取消的任务必须支持取消。
- 页面销毁或状态切换时，应取消不再需要的任务。
- 避免重复启动相同任务。
- 使用 `Task`、定时器、通知监听或 Combine 订阅时，必须考虑释放和取消。
- 禁止使用无限循环模拟刷新。
- 必须避免数据竞争、重复回调和状态错乱。

### 4. 内存管理

- 避免循环引用。
- 闭包捕获对象时，根据生命周期合理使用 `weak` 或 `unowned`。
- 图片、歌词、文件列表等数据不得无限缓存。
- 大图片应按实际展示尺寸加载或缩放。
- 文件缩略图应按需生成，并设置合理缓存上限。
- 列表中不可见项目不应保留复杂视图状态。
- 对象释放后，对应监听、定时器和任务必须停止。

### 5. CPU 控制

- 禁止使用毫秒级定时器持续刷新普通 UI。
- 时间显示通常每秒刷新一次即可。
- 歌词进度应根据实际播放状态更新，不应无条件高频刷新。
- 鼠标未移动、窗口未展开、数据未变化时，不执行重复计算。
- 拖拽、滚动和动画过程只更新必要区域。
- 避免在 SwiftUI `body` 中执行复杂计算、文件读取或数据转换。
- 对高频回调使用节流或防抖，但要保证交互响应。

## UI 与交互要求

- 视觉设计保持简洁、克制。
- 使用自然的圆角、阴影、模糊和层级关系。
- 避免过度发光、过度渐变和明显的 AI 风格。
- 动画应自然，不拖沓，不突兀。
- 所有可交互元素都应有明确反馈。
- 拖拽时必须显示当前拖拽对象和目标状态。
- 成功、失败、不可操作等状态要有明显区别。
- 不允许因为动画阻塞用户连续操作。
- 支持浅色和深色模式时，应分别检查可读性和对比度。
- 文本过长时应合理截断、滚动或换行，不得挤压关键操作区域。

## 文件管理要求

涉及文件功能时：

- 文件移动、复制、重命名和删除必须有明确结果。
- 不允许静默覆盖已有文件。
- 操作失败时必须给出具体原因。
- 拖拽过程应区分重新排序、移动到文件夹和发送到其他区域。
- 大量文件加载应分页、懒加载或分批处理。
- 文件缩略图不得一次性全部同步生成。
- 文件扫描不得阻塞主线程。
- 注意沙盒权限和 Security-Scoped Bookmark 的生命周期。
- 不允许访问用户未授权的目录。
- 文件路径显示过长时，不得影响搜索、排序和其他主要功能。

## 代码修改规则

每次修改代码前：

1. 先确认当前实现和调用链。
2. 明确本次修改范围。
3. 判断是否影响其他功能。
4. 优先进行最小范围修改。
5. 保持现有代码风格。
6. 不修改与当前需求无关的文件。

每次修改完成后：

1. 检查是否存在编译错误。
2. 检查主线程阻塞风险。
3. 检查内存泄漏风险。
4. 检查定时器、通知和异步任务是否正确释放。
5. 检查窗口展开、收起、切换和关闭状态。
6. 检查重复操作是否会产生重复任务或重复监听。
7. 检查异常和空数据场景。
8. 简要说明修改内容和影响范围。

## 输出要求

提供代码时：

- 优先给出可以直接使用的完整代码。
- 明确说明代码应放在哪个文件或位置。
- 不省略关键逻辑。
- 不使用大量伪代码代替实现。
- 不为了展示而添加无关功能。
- 修改现有代码时，应明确指出新增、删除和修改的部分。
- 无法确定上下文时，先要求提供相关文件或代码。
- 不确定 API、系统版本或框架行为时，不要编造结论。

## 禁止事项

- 禁止为了视觉效果持续占用高 CPU。
- 禁止在主线程执行文件扫描、网络请求或大量数据处理。
- 禁止无上限缓存。
- 禁止遗留未释放的定时器和监听器。
- 禁止在需求不明确时自行扩大功能范围。
- 禁止引入不必要的架构和第三方库。
- 禁止修改与当前任务无关的代码。
- 禁止用临时补丁掩盖状态管理问题。
- 禁止忽略错误处理。
- 禁止未经确认删除、覆盖或迁移用户文件。

## 决策优先级

出现冲突时，按以下优先级处理：

1. 数据安全和用户文件安全。
2. 稳定性。
3. 内存与 CPU 占用。
4. 交互正确性。
5. 代码可维护性。
6. 动画和视觉效果。
